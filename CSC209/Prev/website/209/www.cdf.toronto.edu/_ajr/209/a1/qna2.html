Assignment one: KEEP IT SIMPLE!

The more complicated program is less likely to be correct, and all the extra complexity is a negative thing to the users, not a bonus. And it certainly doesn't give you extra marks; whereas the extra bugs you introduce will lose you marks.

Here are some ways in which some people are making the assignment one programs more complicated than they need to be:

  • excessive storage (e.g. variable assignment). If you are thinking of a value such as "$*", you can just use that. Don't assign it to some other variable name. The other variable name is less meaningful (at least, to someone who knows the sh programming language), and the additional copying sometimes introduces bugs, depending on how you do it. Similarly, don't store the output of a command in a variable and then do "echo $variable" -- just run the command with its output going naturally to the standard output (assuming you don't need this output for an additional purpose later).
  • Don't look up the ASCII character values for characters in C. If you want the ASCII character value for 'a', write 'a', not 97. This is clearer to the reader, less error-prone, and so on. Characters in single quotes in C are int values. This includes '\n' for newline and '\t' for tab.
  • Don't use $? in your shell scripts. The 'if' statement tests the success or failure of a command. Test it directly, not indirectly by getting the exit status via $? and then running another command ("test") and using 'if' on that. Just use 'if' on the original command directly. i.e. "if ispalindrome ..."
  • Omit empty 'else' clauses. Similarly, your entire shell script should not be enclosed in an additional set of parentheses. In general, omit anything which doesn't do anything and doesn't say anything.
  • Follow the algorithms in the assignment handout. Other approaches are more complicated, and also in some cases teach you less.

Format your programs with care; don't make a mess. Proper indentation, for one, is a requirement (both in school and in "real life"). Use standard formatting in general; there are better ways to be creative than making your programs difficult to read. And a tab character means eight columns; you might change the tab setting in your editor, but that doesn't change it in everyone else's, so you don't want to change it in yours.
It's harder to make sh programs look nice, and you don't have to do as thorough a job there, but you do have to try a bit, and you certainly have to use standard indentation.

Do not write bash-specific code. Test your shell scripts with both bash and dash.

Also, don't write comments which explain the sh or C syntax to the reader. A comment such as 

# assign 3 to i
i=3
is a negative thing, for the reader and for the author. The reader already knows what "i=3" means. The author has additional text to maintain. If you want to read more about this you could read comments.html (although that's mostly about C, not sh).

And avoid block comments with the date, faux version numbers, etc, especially when this block comment is bigger than your actual program.
